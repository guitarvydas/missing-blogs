# 2023-05-01-Type Checking LiteTo me, the goal is: deliver robust, bug-free, guarantee-able software to the User with the least amount of expense and development time possible.

ATM, that appears to mean: use RAD and dynamic typing to "get into the ballpark" as quickly as possible, i.e. build MVPs rapidly and cheaply, throwing previous versions away whenever you change your mind.  Then Production Engineer.  There is no need for full-blown static type checking until the Production Engineering step.  Type checking is but a crutch for Developers.  One should have access to type-checking-lite during RAD development. I don't know what that means, but, I do know that JavaScript could have saved me countless hours if only it knew how to count arguments (regardless of type).

The final app is not without at least some dynamic checking ("input validation").  Some dynamic type checks might be removed by a compiler.  Why not express ALL type checks in the same way and let the compiler earn its living?  FBP and componentization allows these kinds of thoughts, whereas bolting static type checking into a programming language tends to inhibit these kinds of thoughts.  Worse, embedded type checkers expect Developers to suffer and tweak code to fit what the type checker is capable of checking.  This kind of tweaking inhibits RAD.
