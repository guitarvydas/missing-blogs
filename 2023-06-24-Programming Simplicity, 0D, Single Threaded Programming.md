# 2023-06-24-Programming Simplicity, 0D, Single Threaded Programming
The fact that CPUs use a hardwired, shared, global stack means that hardware designers think of CPUs differently from how software designers have spun the concepts.  I would argue that going back to using CPUs as bits of hardware candy will bring simplicity back to programming.  Hardware CPUs are *meant* to be single-threaded.  Using operating systems on hardware candy creates accidental complexity (aka epicycles).  There is no need for Linux, Windows, MacOS, ROS, etc.  In the 1950s, CPUs were very expensive, so time-sharing made sense.  In 2023, CPUs are very cheap and time-sharing CPUs doesn't make sense anymore.  If you need multiple tasks, use multiple CPUs, each running one task.  If you insist on faking multi-processing on single computers (regardless of how many CPUs are involved), then the 1950s time-sharing model is still the wrong model for multi-tasking.  The better model is like Actors sending messages to each other on pairs of one-way wires.  I call this model "0D".  Modelled this way, the problem devolves into the old GOTO problem - how do you prevent a design from becoming an ad-hoc mess of confusing detail?  The answer is obvious - Structured Message Passing.  Imagine how a successful business is built in a scalable manner.
